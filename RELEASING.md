# Releasing Syng

This document is the source of truth for how Syng is versioned and released.

## Version locations

The release version lives in the following files. **Update all of them together** to the same
value when cutting a release:

1. `package.json` — `version`
2. `src/native/Cargo.toml` — `[package] version`
3. `src/native/tauri.conf.json` — `version`
4. `src/native/gen/apple/project.yml` — `CFBundleShortVersionString` **and** `CFBundleVersion`
   (iOS). Note: `gen/apple/` is generated by `cargo tauri ios init`; if you regenerate it, re-apply
   this version.

Android `versionName` / `versionCode` are derived automatically from `tauri.conf.json` via the
generated `tauri.properties` — **do not** hand-edit `gen/android/app/build.gradle.kts`.

These four files are the source of truth and are updated by hand at release time. This is
deliberately manual: a single grep of `2.0.0` across them confirms consistency, and there is no
sed-based bumper to drift or mangle the differing JSON / TOML / YAML / plist formats.

## `latest-release.json` (auto-updater manifest)

`latest-release.json` is the manifest the desktop updater polls. It reflects the **latest
published release**, which is intentionally *not* the same as the version currently in development.

- It is **not** bumped as part of normal development or this release-prep work.
- It is updated **only when a release is actually published**, by hand (kept deliberately simple —
  no automation, which would be more fragile than maintaining one small JSON file).
- When `2.0.0` ships, bumping this manifest is what delivers the update to existing beta users.

## Release flow

### Desktop (existing channels — the "ship 2.0.0 now" path)

```
make release-desktop      # macOS (both arches) + Windows + Linux (amd64 + arm64)
```

Individual targets: `release-macos`, `release-windows`, `release-linux`. These produce the
signed-updater artifacts consumed by `latest-release.json`.

Publishing steps:
1. Bump the version in all four locations above.
2. Update `CHANGELOG.md` (`[2.0.0] - <date>`).
3. Build with `make release-desktop`.
4. Create the GitHub release, upload artifacts.
5. Update `latest-release.json` to the new version and the artifact URLs.

## Distribution-channel setup (deferred — not required for the desktop binary)

These need external accounts / secrets and are tracked separately from the binary. The in-repo
build scaffolding exists; only submission is deferred.

### Mac App Store (MAS)

MAS builds must exclude the self-updater (Apple forbids self-updating apps). This is handled by the
`mas` Cargo feature + a config overlay:

```
make release-mas          # cargo tauri build --features mas --config tauri.mas.conf.json
```

`tauri.mas.conf.json` nulls the updater plugin and points at `entitlements.mas.plist`
(app-sandbox + network client + user-selected files). The `is_mas_build` command hides the in-app
Updates UI. Still required before submission: Apple Developer Program membership, a Mac App
Distribution certificate + provisioning profile, and an `app-store-connect` upload.

**What to test specifically in a MAS build:** global keyboard shortcuts (blocked under the sandbox
— confirm graceful degradation), reader web fetch (needs `network.client`), file import/export
dialogs (needs `files.user-selected`), app-data persistence in the sandbox container, and that the
Updates UI is absent.

### iOS App Store

`Info.plist` carries `ITSAppUsesNonExemptEncryption = false` (standard HTTPS/OS crypto only).
`gen/apple/ExportOptions.plist` uses `method = debugging` for `make run-ios-device` sideloading;
App Store submission needs a separate export with `method = app-store-connect` plus a distribution
cert + provisioning profile.

### Android (Google Play)

Deferred: Play Console account, an upload/signing keystore wired into a `signingConfig` in the
`release` block of `gen/android/app/build.gradle.kts`, and Play App Signing enrollment.

### Windows (code signing / MS Store)

Deferred: an Authenticode certificate (`bundle.windows.certificateThumbprint`) and, for the MS
Store, store packaging. Unsigned binaries still build and run.

### Dependency license scan (2026-07-01)

**JS production dependencies** (`npx license-checker --production --summary`): all `MIT`,
`Apache-2.0`, or `MPL-2.0`. The single `GPL-3.0` entry is Syng itself. No conflicts.

**Rust crates** (aggregated from `cargo metadata` `license` fields): overwhelmingly
`MIT` / `Apache-2.0` / `BSD` / `ISC` / `Zlib` / `Unicode-3.0`. Notable:

- **MPL-2.0** — `cssparser`, `selectors`, `dtoa-short`, `option-ext` (pulled in by the reader's
  HTML parsing). MPL-2.0 is weak, file-level copyleft, is GPL-3.0-compatible, and imposes no
  anti-DRM / "further restrictions" clause. Fine for both the GPL and app-store distribution.
  No action.
- **`r-efi`** — offered as `LGPL-2.1-or-later OR MIT OR Apache-2.0`; we take MIT/Apache, so no
  copyleft obligation.
- ✅ **EPUB parsing is `rbook` 0.7.9 (`Apache-2.0`)** — the reader previously depended on the
  `epub` crate (`2.1.5`, `GPL-3.0`). Because Preston does not hold that crate's copyright, the
  GPLv3 §7 additional permission in `LICENSE-APP-STORE-EXCEPTION` could not extend to it, which
  left the statically-linked binary blocked from the **Apple App Store / Mac App Store and iOS**.
  It was replaced with `rbook` (`Apache-2.0`) on 2026-07-01. `rbook`'s only added transitive
  crates — `quick-xml` (`MIT`), `typed-path` (`MIT OR Apache-2.0`), `zip` (`MIT`) — are all
  permissive. **No GPL-licensed dependency remains; the sole `GPL-3.0` entry in the tree is Syng
  itself**, so the App Store exception now cleanly covers the whole distributed binary. (MAS/iOS
  are still deferred for the cert / provisioning / submission work noted above, not for licensing.)
